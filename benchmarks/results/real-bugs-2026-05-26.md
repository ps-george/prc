# Real-bugs benchmark — 2026-05-26

A 20-instance comparison of **prc** against a single-call vanilla-LLM
review baseline, on real human-validated bugs sampled from
**SWE-bench Verified**.

## Setup

- **Source:** [SWE-bench Verified](https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified)
  (500 instances, all human-vetted real-world Python bugs).
- **Sample:** 20 instances, stratified by `golden_patch` line-count
  from small (4 changed lines) to medium (41 lines). Repos covered:
  astropy, django, matplotlib, pytest, scikit-learn, sphinx, sympy.
- **Bug injection:** each instance's `golden_patch` is **reversed** to
  produce a *bug-introducing* unified diff — i.e. a synthetic PR that,
  if merged, would re-introduce a real, human-validated bug. The
  reviewer sees only the reversed diff and a short commit subject; it
  does not see the issue text or test names.
- **Model:** `anthropic/claude-sonnet-4.6` via OpenRouter
  (`https://openrouter.ai/api/v1`), same model across all three systems.
- **Systems compared:**
  1. **prc** — `understand → correctness → verdict` pipeline.
  2. **vanilla** — single chat-completions call: *"Please review this
     PR. Did you find any bugs?"*.
  3. **prc-forward-FP** — prc fed the *forward* (real-fix) patch.
     Anything it flags here is a false positive: flagging a known-good
     human fix as a bug.
  4. **vanilla-forward-FP** — same FP control for the vanilla baseline.
- **Total measured cost:** ≈ $2.50 across all four passes
  (well under the $5 cap).

## Scoring

- **prc caught**: the structured verdict contains at least one
  blocker- or major-severity issue whose `location.file` matches the
  bug's file (lenient: basename match accepted).
- **vanilla caught**: the response text contains affirmative
  bug-language (e.g. *"yes, there's a bug"*, *"this will break"*,
  *"the bug is..."*) and does not also conclude *"looks fine"*. The
  file-path is *not* required, because vanilla replies typically
  describe the bug behaviourally without quoting the path. Earlier
  scoring required a path mention and produced an artificial 0% recall
  for vanilla; this is the corrected scoring.

## Headline numbers

| System            | Recall (reversed) | FP rate (forward) | Recall − FP | Wall time | $/successful catch |
| ----------------- | ----------------- | ----------------- | ----------- | --------- | ------------------ |
| prc               | **90% (18/20)**   | 60% (12/20)       | 0.30        | 389 s     | ~$0.07             |
| vanilla           | **100% (20/20)**  | 65% (13/20)       | 0.35        | 146 s     | ~$0.02             |

Notes:
- *Recall − FP* is a quick eyeball — it's not a real precision figure
  (precision can't be computed from this design because we don't have
  per-issue ground truth, only file-level).
- prc costs ≈ 3× vanilla per instance (two LLM calls vs one), and is
  ≈ 2.7× slower wall-clock.

## Honest reading

**On this benchmark, the vanilla control beats prc.** Vanilla achieves
strictly higher recall (100% vs 90%) at strictly lower cost and
latency. prc's only edge is a marginally lower false-positive rate
(60% vs 65%) — not enough to offset the 2-instance recall gap or the
3× cost.

The bigger story is that **both reviewers have high false-positive
rates** on this benchmark design (~60-65%). The reversed-patch
construction creates a strong "code was removed, are you sure?" signal
that biases any reviewer toward flagging something. Conversely, when
shown the real fix in the forward direction, both reviewers
*incorrectly* call out 12-13 of the 20 fixes as bugs. The benchmark
is, in this sense, easy on recall and brutal on precision for any
verbose reviewer.

**What this benchmark cannot tell us:**
- whether prc is better/worse than CodeRabbit, Greptile, etc. — those
  weren't run (no free API tier accessible).
- whether prc helps on larger or multi-file changes — the sample
  caps at 41 changed lines.
- precision against a per-issue ground truth — we only score
  file-level catches.

## Per-instance results

### Vanilla caught, prc missed (2 instances)

These are the only instances where the pipeline lost a recall point to
the single-call baseline.

| Instance | Bug file | prc decision | vanilla snippet |
| --- | --- | --- | --- |
| `django__django-16493` | `django/db/models/fields/files.py` | approve (no blockers) | "The refactored code has a bug. In the original code, `storage` is resolved via `getattr(self, "_storage_callable", self.storage)` **before** the comparison..." |
| `astropy__astropy-14096` | `astropy/coordinates/sky_coordinate.py` | approve (no blockers) | "...there is a subtle bug/regression: `self.__getattribute__(attr)` would raise `AttributeError` too, but it also invokes the normal attribute lookup machinery..." |

In both cases the bug is subtle and the vanilla reviewer happens to
reason its way through it in prose; prc's correctness phase returned
an empty issue list, so the verdict phase auto-approved.

### prc caught and *did not* false-positive on the matching forward fix (7 instances)

These are the instances where prc shows the cleanest signal —
flagged the bug on the reversed diff *and* let the real fix through:

- `django__django-15375`  (`django/db/models/aggregates.py`)
- `sympy__sympy-21847`    (`sympy/polys/monomials.py`)
- `django__django-14608`  (`django/forms/formsets.py`)
- `sympy__sympy-19954`    (`sympy/combinatorics/perm_groups.py`)
- `scikit-learn__scikit-learn-12973` (`sklearn/linear_model/least_angle.py`)
- `django__django-15732`  (`django/db/backends/base/schema.py`)
- `astropy__astropy-14182` (`astropy/io/ascii/rst.py`)

### Vanilla caught but vanilla also false-positives on the forward fix

Vanilla's 100% recall comes paired with a 65% forward-fix FP rate;
on roughly two-thirds of cases the vanilla baseline is "always
suspicious", which means many of its 20 "catches" are difficult to
distinguish from background noise.

## Position

A single 20-instance benchmark on one bug family (reversed
SWE-bench Verified patches) does *not* tell us prc is better or worse
than commercial PR-review tools in general. It tells us one specific
thing: **on these 20 reversed-patch instances and this exact prompt,
prc's two-phase pipeline does not improve recall over a single
"did you find any bugs?" call to the same model, and costs ~3× more.**

The case for prc's structured output (severity, category, location,
explicit verdict) is *not* tested by this benchmark — recall on a
"yes/no any bug?" question is the wrong dimension to evaluate
structure on. A benchmark that does test that would compare
*actionability* (e.g. can the author respond in-protocol? does the
verdict help triage?) and is left for a later round.

## Reproducing

```bash
cd benchmarks
export PRC_MODEL=anthropic/claude-sonnet-4.6
export PRC_API_KEY=$OPENROUTER_API_KEY
export PRC_API_BASE=https://openrouter.ai/api/v1
uv run python -m benchmarks.run_real_bugs
```

The sample file `benchmarks/data/sample_20.json` is checked in;
regenerating it requires the `datasets` package and a network
connection to Hugging Face.
